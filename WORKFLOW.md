# 分身LLM開発ワークフロー(学習記録)

自分の全会話データから「自分の分身LLM」を作るプロジェクトの全工程。
各フェーズに業界での正式名称を併記(ES・面接・技術記事用)。

## フェーズ0: 基礎理解 — LLMをゼロから作る
**業界用語: 事前学習(pretraining)の縮小版 / 言語モデリング**

- `mini_gpt/` にて、PyTorchで文字レベルGPT(1.66Mパラメータ)をフルスクラッチ実装
- 学んだ部品: トークナイザ(文字→ID)、埋め込み、自己注意機構(Q/K/V)、
  位置埋め込み、残差接続、LayerNorm、学習ループ(予測→損失→逆伝播→更新)
- 実験1: 青空文庫55万文字で学習(検証損失 8.16→3.87)
- 実験2: 自分の文章12.5万文字で再学習 → **過学習をライブ観測**
  (訓練損失は下がり続けるが検証損失が iter 1600 で反転)
  → 対策としてベスト検証損失時のみ保存(early stopping の考え方)を実装
- 得た直感: 「LLM=次のトークン予測」「データ量と記憶容量のバランスが汎化を決める」

## フェーズ1: データ収集
**業界用語: データ収集(data collection)/ データソーシング**

| ソース | 取得方法 | 収穫 |
|---|---|---|
| LINE(10人分) | トーク履歴を送信(公式エクスポート) | 690件の実会話 |
| ChatGPT | 設定→データエクスポート(6分割JSON) | 1,820ペア |
| Claude | Settings→Export data | 445ペア |
| Gemini | Google Takeout「マイ アクティビティ」JSON | プロンプト1,065件(文体用) |
| 紙の日記30枚 | 写真→Claudeが文字起こし(手書きOCR) | 約2.9万文字 |

- 教訓: エクスポート形式はサービスごとにバラバラ(ツリー構造/フラット/操作ログ)。
  Geminiは「Gemini」ではなく「マイ アクティビティ」に会話が入っている等、罠が多い

## フェーズ2: データ整形
**業界用語: データキュレーション / 前処理(preprocessing)/ SFTデータセット構築**

`bunshin_data/make_dataset.py` が全自動で実行:

1. **形式の自動判別** — ChatGPT(mapping木構造)/Claude/Gemini を中身で識別
2. **クリーニング** — スタンプ・通話記録・URL単体・コード貼り付けを除去
3. **セッション分割** — 60分以上空いたら別会話(昨夜の「おやすみ」と今朝の「おはよう」を混ぜない)
4. **ターン結合** — 同一人物の連続メッセージを1ターンに
5. **関係ラベル付け** — `partners.json` で相手→関係(家族/友達/先生)を全例に付与
   → 分身が「相手によって話し方を切り替える」ための条件付け(conditioning)
6. **文脈つき学習例化(v2)** — 返事1つごとに直前6ターンの流れを付ける(マルチターン化)
7. **ソース比率調整(v2)** — AI由来が77%と偏っていたため800件に間引き(リバランス)
8. 出力: チャット形式JSONL(`messages`+`source`+`relation`)= 世界標準のSFT形式

## フェーズ3: プライバシー処理
**業界用語: PII仮名化(pseudonymization)/ データガバナンス**

- `anonymize.py` + `name_map.json`: 実名743カ所を自然な仮名に置換
  (妹・友達・先生の名前、LINE表示名、名字のローマ字/ひらがな表記ゆれまで)
- 置換後に「元の名前が1つでも残っていないか」自動検証
- 運用ルール: クラウド(Kaggle/Colab)に上げるのは仮名版のみ。
  実名データ・日記原文・学習済みモデルは非公開・ローカルのみ

## フェーズ4: ファインチューニング
**業界用語: SFT(教師ありファインチューニング)/ QLoRA / PEFT**

- ベースモデル: Qwen3.5-4B → 9B(3号)。事前学習済みの日本語力を「借りて」、
  自分らしさだけを追加学習する転移学習
- **QLoRA**: モデル本体は4bit量子化で凍結、小さな差分アダプタ(LoRA, r=16)のみ学習
  → 無料GPU(Kaggle T4)で学習可能に。過学習も抑制
- **ペルソナ条件付け**: 各例のsystemプロンプトに「あなたは息吹。相手は{関係}」を注入
- **train_on_responses_only**: 相手の発言は損失計算から除外し「自分の返事」だけ学習(損失マスキング)
- 検証分割5% + 損失カーブ監視(フェーズ0で学んだ読み方をそのまま適用)
- インフラの教訓: Colab無料枠(Drive接続失敗・GPU枠)→ Kaggle(週30h)に移行。
  「学習したら即保存」の順序、OOM時のフォールバック

## フェーズ5: 評価と反復(イテレーション)
**業界用語: 評価(evaluation)/ エラー分析 / 反復開発**

| 版 | 構成 | 結果と診断 |
|---|---|---|
| 1号 | 4B + 単発ペア2,955組 | 口調・方言は再現、返事が支離滅裂 → 原因: 文脈ゼロのデータ |
| 2号 | 4B + 文脈つき | 関係による切替・LINE的な複数送信を再現。短文挨拶に脊髄反射 / 検証損失2.50でまだ下降中 → 診断: テスト条件のミスマッチ + 学習不足(underfitting) |
| 3号 | 9B + 5周 + 連続会話評価 | 学習中 |

- 学び: 「過学習」と「学習不足」を両方、自分の損失カーブで観測した
- 学び: 評価は学習分布と揃える(マルチターンで学習したなら会話を続けて評価する)
- 予定: 固定10問の比較評価 + 家族・友達による「どっちが本人?」ブラインドテスト

## フェーズ6(予定): デプロイ
**業界用語: 量子化(quantization)/ ローカル推論 / エッジデプロイ**

- GGUF形式(q4_k_m, 約2.5GB)に変換 → Ollamaで自分のPC上でオフライン動作
- ゴール: 面接の場でWi-Fiなしのライブデモ

## この経験が対応する実務
機械学習エンジニアリングのライフサイクルそのもの:
**データ収集 → 前処理 → ガバナンス → 学習 → 評価 → 改善ループ → デプロイ**
(いわゆるMLOpsの一周。多くの企業で「データ整形と評価が仕事の8割」と言われる部分を全部自分の手でやった)
